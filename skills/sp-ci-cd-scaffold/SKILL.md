# Superpowers: CI/CD Scaffold

This skill bootstraps the CI/CD + E2E baseline that every later feature slice depends on. It is the canonical Stage 1 of any phased plan generated by `prd-to-phased-plans`. It runs end-to-end on a dedicated `chore/ci-cd-scaffold` branch and ends with a green PR merged to `main`.

This skill writes infrastructure, not product code. After it completes, `sp-feature-delivery` (driven by `the-orchestrator` or invoked directly) can safely begin shipping features.

## Inputs and Preconditions

- Repository exists with at least one app or package directory.
- Clean git working tree on `main`, OR explicit user OK to proceed dirty.
- `gh` CLI installed and authenticated (needed for PR + branch protection).
- `pnpm` (or detected alternative: `npm` / `yarn` / `bun`) installed.

If the repo already has all four scaffold artifacts present (`.husky/pre-push`, `.github/pull_request_template.md`, `.github/workflows/e2e-coverage.yml`, `scripts/setup-branch-protection.sh`), stop and tell the user the baseline is already in place; recommend `/sp-feature-delivery` instead.

## Workflow

### Phase 0 — Discovery (orchestrator only)

1. Detect package manager: `pnpm` (preferred) → `npm` → `yarn` → `bun`. Inspect lockfile.
2. Detect app framework/runtime (Next.js, Vite/React, Node API, etc.).
3. Detect monorepo tooling: Turborepo (`turbo.json`), Nx (`nx.json`), or single-app.
4. Inventory existing `.github/workflows/*.yml` and any test tooling already present.
5. Identify target app(s) for E2E smoke + regression coverage.

### Phase 1 — Clarifying Questions (Plan Mode Required)

Before any write, switch to **Plan Mode** and ask at most 5 focused questions:

1. Which app(s) are critical paths for E2E?
2. Should CI run on PR only, or PR + push to `main`?
3. Which suites are required blockers (`@feature`, `@regression-core`, full)?
4. Are deployments gated on green E2E checks?
5. Any restricted environments/secrets needed in CI?

Wait for answers before continuing.

### Phase 2 — Branch Setup

- Create a dedicated branch: `chore/ci-cd-scaffold`.
- Never scaffold CI directly on `main` / `master`.
- Keep this PR minimal: baseline quality + smoke E2E only. No product features in this slice.

### Phase 3 — Implementation (sequential)

Implement in five sub-blocks. Each ends with a commit before moving to the next. Use the canonical content in [references/scaffold-artifact-templates.md](references/scaffold-artifact-templates.md) for every file you write.

#### A. Test framework & E2E suites

1. Install Playwright (or detected E2E framework).
2. Add scripts to root `package.json`:
   - `test:e2e`
   - `test:e2e:feature`
   - `test:e2e:regression`
3. Add baseline tests:
   - smoke spec tagged `@feature`
   - regression sentinel tagged `@regression-core`
4. Extend the task runner config (e.g. `turbo.json`) for the new E2E tasks if a monorepo runner is present.

#### B. CI workflows

5. Add `.github/workflows/ci.yml` — install, lint, typecheck, unit/integration tests.
6. Add `.github/workflows/e2e.yml` — install, browser deps, run `@feature` and `@regression-core`, upload traces/videos/reports on failure.
7. Add `.github/workflows/e2e-coverage.yml` — single job that diffs PR vs `main` and fails if any file under `apps/**` or `packages/**` was changed without a corresponding `*.spec.ts` or `*.e2e.ts` change in the same PR. Exempt: `*.md`, `*.json` (config-only), `*.css`, image/font assets. Job name: `E2E / coverage-check`.
8. Update `.gitignore` for E2E artifacts (`playwright-report/`, `test-results/`, `.playwright/`).

#### C. Husky pre-push hook

9. Install Husky: `pnpm add -D husky && pnpm exec husky init`.
10. Write `.husky/pre-push` with the local gate chain — see [references/scaffold-artifact-templates.md](references/scaffold-artifact-templates.md).
11. Commit `.husky/pre-push` so every contributor and agent gets the hook automatically.

#### D. PR template

12. Generate `.github/pull_request_template.md` with the E2E attestation checklist — see [references/scaffold-artifact-templates.md](references/scaffold-artifact-templates.md).

#### E. Branch protection provisioning

13. Generate `scripts/setup-branch-protection.sh` (uses `gh` CLI). Required status checks:
    - `CI / lint`
    - `CI / typecheck`
    - `CI / unit-tests`
    - `E2E / feature`
    - `E2E / regression-core`
    - `E2E / coverage-check`

    The script is run **once** by the project owner after repo creation. It is not executed automatically by this skill.

### Phase 4 — Local Verification Gates

Before opening the PR, run all of these and confirm green:

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test` (unit/integration)
- `pnpm test:e2e:feature`
- `pnpm test:e2e:regression`

If any fail, fix on the same branch before proceeding. The Husky `pre-push` hook will enforce these on push automatically.

### Phase 5 — PR + CI/CD

1. Push `chore/ci-cd-scaffold` to remote.
2. Open the PR via the `git-commit-push-pr` or `new-branch-and-pr` skill if available; otherwise `gh pr create`. PR description must list every artifact created.
3. Wait for CI to complete. If any required check fails, patch on the same branch and push until green.
4. Once CI is green, merge.

### Phase 6 — Closeout

1. After merge, update local `main`: `git checkout main && git pull --ff-only origin main`.
2. Delete the local branch: `git branch -d chore/ci-cd-scaffold`.
3. Delete the remote branch if not auto-deleted: `git push origin --delete chore/ci-cd-scaffold`.
4. Verify `git status` is clean on `main`.
5. Remind the user (once) to run `scripts/setup-branch-protection.sh` to enable required status checks on `main`.
6. Walk [references/scaffold-completion-checklist.md](references/scaffold-completion-checklist.md) end-to-end and confirm every box is `[x]`.

## Subagent-Oriented Execution

When Phase 3 work is non-trivial (e.g. complex monorepo wiring), prefer:

1. One implementer subagent per independent sub-block (A–E).
2. Spec-compliance review after each sub-block.
3. Code-quality review after spec-compliance passes.
4. Fix review findings before moving to the next sub-block.

Reuse the same subagent pattern documented in `sp-feature-delivery` if available.

## Final Output Format

After Phase 6, report to the user:

1. Files created / updated (grouped: workflows, husky, pr template, scripts, tests, package.json scripts).
2. CI triggers and required status checks.
3. Local verification commands and outcomes.
4. Confirmed presence of all four scaffold artifacts:
   - `.husky/pre-push`
   - `.github/pull_request_template.md`
   - `.github/workflows/e2e-coverage.yml`
   - `scripts/setup-branch-protection.sh`
5. PR URL and merged commit SHA.
6. Recommended next E2E flows to add (handed to `sp-feature-delivery`).
7. Reminder: run `scripts/setup-branch-protection.sh` once to enable required status checks on `main`.

## Hard Constraints

- **Never scaffold CI directly on `main` / `master`.** Always use `chore/ci-cd-scaffold`.
- **Never weaken or remove existing workflows.** This skill adds; it does not subtract. If existing workflows conflict, surface to the user and stop.
- **Completion checklist is mandatory.** At the end of every run, walk [references/scaffold-completion-checklist.md](references/scaffold-completion-checklist.md) end-to-end and confirm every box is `[x]`. The scaffold is not "done" — and the orchestrator must not report success — until all six sections of that checklist pass.
- **Stage 1 contract.** When invoked as Stage 1 of a phased plan, this skill is the entire stage. The plan file at `docs/plans/stage_1_ci_cd_scaffold.md` delegates to this skill and lists no separate task body. After completion, mark the stage `Completed` in `docs/plans/00_master_checklist.md`.

## Triggers

Follow this skill whenever the user:

- runs `/sp-ci-cd-scaffold`
- says "scaffold ci/cd", "set up CI", "bootstrap quality gates", "set up Playwright + GitHub Actions"
- begins Stage 1 of a phased plan generated by `prd-to-phased-plans`
- adds a new repo that needs the baseline

If the repo already has the baseline, stop and recommend `/sp-feature-delivery` instead.
