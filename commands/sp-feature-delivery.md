# Superpowers: Feature Delivery

Use this command to execute phased implementation work generated by `/prd-to-phased-plans`.

## Primary Contract

Treat `docs/plans/` as the source of truth:
- `docs/plans/00_master_checklist.md` is the execution tracker.
- `docs/plans/stage_<n>_*.md` is the implementation spec for each stage.
- If checklist and stage file conflict, ask the user before proceeding.

## Auto-Discovery Flow

1. Find and read:
   - `docs/plans/00_master_checklist.md`
   - all `docs/plans/stage_*.md`
2. Determine current active stage by:
   - first stage marked `Not Started` or `In Progress` in checklist, or
   - explicit user-selected stage.
3. Build an execution map:
   - stage -> feature -> checklist items -> candidate tasks.

## Plan Mode First (Mandatory)

Before coding, switch to Plan Mode and ask focused alignment questions:
1. Which exact checklist items are in-scope now?
2. Which items are explicitly out-of-scope for this pass?
3. Is this pass MVP-only or can it include Phase 2 work?
4. What are acceptance tests for this slice?
5. Any constraints on UX/API/data behavior that must not change?

Then produce a short, dependency-ordered implementation plan mapping each task to checklist checkboxes.

## Branch and Worktree Strategy

Always isolate each delivery slice:
- branch naming:
  - `feat/stage-<n>-<scope>`
  - `fix/stage-<n>-<scope>`
  - `chore/stage-<n>-<scope>`
- prefer a git worktree for each active slice.
- never implement directly on `main`/`master`.
- keep one checklist slice per PR when possible.

## Subagent-Driven Orchestration

Execute tasks in `/subagent-driven-development` style:
1. One implementer subagent per task.
2. Then spec-compliance review subagent.
3. Then code-quality review subagent.
4. If either review fails, send back to implementer and re-review.
5. Mark task complete only after both reviews pass.

Never run parallel implementer subagents on the same branch.

## Test and Quality Gates

For each checklist slice, enforce:
1. Local gates before push:
   - lint
   - typecheck
   - unit/integration tests
   - affected E2E tests
2. E2E structure:
   - `feature` suite for new/changed behavior
   - `regression-core` suite for critical existing flows
   - rerun impacted historical suites when shared components/routes change
3. Never mark checklist items done until tests are green.

## CI/CD Guardrails

If missing, scaffold baseline CI/CD:
- run `/scaffold-ci-cd` first to create the baseline pipeline and E2E structure
- `.github/workflows/ci.yml` for lint/type/test on PR
- `.github/workflows/e2e.yml` for feature + regression-core
- artifact upload on E2E failure (trace/video/report)
- require checks before merge

If baseline already exists, extend it for new slices:
- add/adjust E2E tags, suites, and paths for newly implemented flows
- widen regression coverage when shared modules/routes are touched
- keep required checks aligned with the same local gates used by this command

## Checklist Update Rules

Update `docs/plans/00_master_checklist.md` continuously and truthfully:
- `[ ]` -> `[x]` only after verified passing tests
- status transitions:
  - `Not Started` -> `In Progress` when first task starts
  - `In Progress` -> `Completed` only when all stage checklist items pass
- add concise implementation notes when scope or approach deviates from stage plan

## Progress Reporting Format

After each task, report:
1. Checklist items completed
2. Files changed
3. Tests run and results
4. Open risks/blockers
5. Next recommended checklist slice

## Seamless Operation With /prd-to-phased-plans

When `docs/plans/` exists, do not ask broad setup questions already answered by the plan set.
Only ask for clarifications on:
- scope boundaries,
- trade-off decisions,
- ambiguous acceptance criteria,
- conflicting instructions between checklist and stage file.
